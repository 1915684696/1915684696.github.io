git首次安装的配置

首次使用git,需要进行如下配置：打开cmd命令行，输入：
git config –global user.name “用户名”
git config –global user.email “邮箱”

注意：这里的用户名和邮箱要用英文的
让它知道主人是谁，以后每次的提交都会用到这些信息

输入完成之后，在命令行中输入git config --list可以可以查看配置后的信息，若显示了name和email就说明配置成功了

git理论基础

git记录的是什么？
git记录的是每次版本的变动，原理则是将每个版本独立保存，它每个版本都是独立的存储，它看似消耗了很多空间，但是在分支管理上却带来了很多的益处和便利。
git实现版本控制的方法？

在git中有这样的三棵树：工作区、暂存区、仓库。
工作区就是我们通常使用的目录。
暂存区存放用户临时的改动，它本身是一个文件，保存的是即将提交到仓库的一个列表信息。
仓库是最终存放所有版本的位置，它有一个head指针，指向最新提交的内容。

git的工作流程和实战

git的工作流程一般是酱紫的：

  1. 在工作目录中添加、修改文件
  2. 将需要进行版本管理的文件放入暂存区域
  3. 将暂存区域的文件提交到git仓库

git管理的文件有三种状态：
——已修改（modified）
——已暂存（staged）
——已提交（committed）

git实现一个版本控制的基本操作如下

1. 创建一个新的文件夹，打开cmd进入文件夹，
2. git init    //初始化仓库
3. 在工作目录中创建文件README.md
4. git add README.md    //添加文件到暂存区
5. git commit -m "你干了啥"    //添加到仓库

git查看工作状态和历史提交

git status    //该命令可查看当前的git的状态，在哪个分支，提交的信息等
git reset HEAD [filename]    //将暂存区的文件恢复回去，filename不写代表恢复最后一次的add
git checkout -- filename        //将修改过的文件退回到之前add过的那个版本  
git    log    //查看历史提交（只显示当前分支），按照时间顺序从进到远

回到过去

git是一个设计精良的时间机器，可以在不同的版本之间来回穿梭

git    reset HEAD~    //~表示退回到上一个版本,~~表示上上一个快照
git reset HEAD~num    //表示退回到head之前的第num个版本

reset命令的选项

git reset --mixed HEAD~        //默认的mixed，可以不写
    --移动HEAD的指向，将其指向上一个快照
    --将HEAD移动后的指向的快照回滚到暂存区域

git reset --soft     HEAD~
    --移动HEAD的指向，将其指向上一个快照

git reset --hard     HEAD~
    --移动HEAD的指向，将其指向上一个快照
    --将HEAD移动后的指向的快照回滚到暂存区域
    --将暂存区域的文件还原到工作目录

git回滚指定快照

git可以回滚到指定的快照，不需要head加上~，可以直接回到指定的版本

git reset "版本号（前几个字母即可）"

git回滚个别文件

git可以回滚快照中的个别文件，在这里HEAD指针不会改变，具体命令如下：

git reset 版本快照 文件名/路径

git往前滚

git不仅可以往回滚，还可以往前滚，就是回到未来：

git reset 版本快照的ID号

git版本对比

git status只能显示版本的是否改动，但不能具体改动了那些地方，改动了多少行，这里的命令可以查看git的具体改动

git diff    //比较暂存区域与工作目录

  • 这里的a/README.md和b/README.md分别代表存放在暂存区的README和存放在工作目录的README，可以当做是git将它们放在了a文件和b文件夹下面进行比较
  • index 1ca24e6...是文件的ID，10064是文件的类型和权限，和linux相同
  • — a/README.md是存放在暂存区中的文件，是旧文件
  • +++ b/README.md是存放在工作目录中的文件，是新的文件
  • 两个@@之间的数字：-代表旧文件，+代表新文件，符号后边的代表开始的行号，数字代表连续的行数
  • 当文档太大显示不完时，会出现类似vi/vim的界面，操作跟vi/vim相同

比较两个历史快照

get diff 快照ID1 快照ID2

比较当前工作目录和Git仓库中的快照

git diff 快照ID

比较当前工作目录和最新提交的内容的快照

git diff HEAD

比较暂存区和git仓库的快照

git diff --cached [快照ID]    //ID没有则指定跟最新的版本进行对比

git修改最后一次提交、删除文件和重命名文件

在实际开发中，可能遇到以下两种情景：

  • 情景一：版本刚一提交（commit）到仓库，突然想起漏掉两个文件还没有添加（add）。
  • 情景二：版本刚一提交（commit）到仓库，突然想起版本说明写得不够全面，无法彰显你本次修改的重大意义。。。。
    这时我们可能比较头疼，但是在git中只要是在本地发生的情况，git基本都可以将它恢复的。。
    执行带 –amend选项的commit提交命令，Git就会“更正”最近的一次提交
git commit --amend    //界面操作跟vi/vim相同

不小心删除文件

使用checkout命令将暂存区的快照恢复到工作目录

git checkout -- 文件名    //注意：--符号的两边各有一个空格

如何彻底删除文件呢？

彻底删除文件可以使用rm命令：该命令删除的只是工作目录和暂存区域的文件，也就是取消跟踪，在下次提交时不纳入版本管理

git rm 文件名    //可以删除当前目录和暂存区的最新快照中的文件，仓库中还会保留
git reset --soft HEAD~    //配合rm命令回到过去，soft只修改仓库指针

若工作目录和暂存区的文件不同（即工作目录已修改），使用rm命令会出现提示信息（因为git不知道你要删除哪一个），但是也可以使用-f命令将两个都强行删除，如下：

git rm -f 文件名字

若只想删除暂存区的文件，但是又不想删除工作目录的文件，可以执行以下操作实现：

git rm --cached 文件名

修改文件名

修改文件名之后，查看git status,git会以为你将原文件删除，又通知增加了一个新的文件，该文件名就会显得很麻烦，它也不是那么智能的。那若想修改文件名，可以在git上直接修改，跟linux相同：

git mv 旧文件名 新文件名        //查看status会显示rename状态

git匿名分支和checkout命令详解

git匿名分支

当使用checkout命令时，后边接上的如果不是分支名，而是HEAD~或者是某个版本的ID时，就会出现：

这是因为使用了checkout命令但是却并没有指明分支名，所以Git创建了一个匿名的分支，但你重新切换到了其他的分支之后，在这个分支里所有进行的操作就会丢失，因为自己并没有为他们命名，切换之后自然也就找不到了。因此我们可以在匿名分支中做一些实验，反正又不影响正常的分支。

git branch 分支名 提供的分支号        //分支号现在还能找到，以后就很麻烦了

git checkout命令详解

事实上，checkout命令有两个功能：

  1. 从历史快照（或者暂存区）中拷贝文件到工作目录
  2. 切换分支
从历史快照（或者暂存区）中拷贝文件到工作目录

当给定某个文件名时，Git会从指定的提交拷贝文件到暂存区域和工作目录，比如：

git checkout HEAD~ README.md    //将仓库的上一个快照中的README.md文件复制到工作目录和暂存区域中

git checkout README.md    //没有指定ID,直接从暂存区覆盖到工作目录

git checkout --READE.me    //这里的文件名前有两个--的命令

注意：–是为了预防你恰好有一个叫做README.md的分支，它就搞不懂你要回复文件还是切换分支了，所以约定两个（–）后边跟的是文件名，若没有一个叫README.md的分支，直接使用git checkout README.md也是可以的

git切换分支

git在切换分支时我们可以以为它做了两件事：

  1. 切换HEAD的指针：将HEAD的指针指向命令中的分支
  2. 将分支中的快照恢复到暂存区域和工作目录

    checkout和reset命令的区别

    那么回过头来，如果我们指向恢复指定的文件/路径，那么我们只需要恢复指定的快照到暂存区域和工作目录，而忽略HEAD指针的移动不就好了么？那样的话不就跟reset命令一样了么？
    这里讲一下checkout和reset命令的区别：
    恢复文件
    checkout命令和reset命令都可以用于恢复指定快照的指定文件，并且它们都不会改变HEAD指针的指向 。它们的区别是：
    reset命令只能指定文件恢复到暂存区域（–mixed），而checkout命令是同时覆盖暂存区域和工作目录。
    注意：也许你视图使用git reset --hard HEAD~ README.md命令让 reset同时覆盖工作目录，但Git会告诉你这是徒劳（此时reset不允许使用–soft或–har选项）。
    这样看来，reset要比checkout命令更安全一些。
    恢复快照
    reset命令是用来“回到过去”的，根据选项的不同，reset命令移动HEAD指针（–soft）、覆盖暂存区域（–mixed，默认）、覆盖工作目录（–hard）。
    checkout命令虽说是用户切换分支，但事实上也是通过移动HEAD指针和覆盖暂存区域、工作目录来实现的。
    主要区别：
  3. 对于reset --hard来讲，checkout命令会更安全。checkout命令在切换分支前会先检查一下当前的工作状态，如果不是“clean”的话，Git不会允许你这样做，而reset --hard则直接覆盖所有数据。
  4. 更新HEAD指向的区别：reset命令会移动HEAD所在分支的指向，而checkout命令只会移动HEAD自身来指向 另一个分支
